Determinism Guarantee

One-line: Same input + same cfg_checksum + cold Q-table → same omega and same recommended_action, byte-for-byte. The narrative "deterministic decision boundary system" holds in the absence of feedback events.

This page explains exactly what that qualification means, how to verify it for your tenant, and what to do if you need stronger guarantees for forensic / replay / compliance work.

What is deterministic by default

For any single POST /v1/preflight call, these components produce identical output for identical input within a single process:

  • Mahalanobis distance — the covariance matrix is computed from the entries in your request only. No rolling window, no cross-tenant population. Same memory_state → same Mahalanobis distance, always.
  • Free energy / M_energy — the formula (F, elbo, kl_divergence, reconstruction) is closed-form over your request payload plus the loaded scoring config.
  • The 10+1 omega components (s_freshness, s_drift, s_provenance, …, r_importance) — pure formulas over the per-entry inputs you send. (For example, s_drift is a fixed linear combination of s_freshness and s_interference with weights from the loaded config.) s_recovery uses the healing_counter you supply per entry, not internal state.
  • The base decision band — omega_final < t_warn → USE_MEMORY, and so on. Thresholds come from the loaded _CFG, which is locked against post-import mutation.
  • Mahalanobis χ² threshold — Wilson–Hilferty closed-form approximation.

In a freshly-restarted process, two back-to-back calls with the same payload return identical responses. We verified this empirically across 30 calls in the determinism audit (3 cases × 10 runs over 60s, zero drift in either omega or decision).

What can drift across calls (and why it usually doesn't)

Three subsystems can override recommended_action based on cross-call state. None of them mutate during a /v1/preflight call — they only evolve when external feedback events occur.

1. Q-table policy gradient override

  • What: a per-domain reinforcement-learning Q-table that learns from /v1/outcome feedback.
  • Trigger to override: a minimum number of episodes for the called domain, a positive advantage above a configured threshold, and not in exploration mode.
  • Mutates when: you call POST /v1/outcome to close a case. Each outcome for that domain accrues one episode.
  • Cold-start protection: below the episode minimum for a domain, the override never fires — base scoring decides.
  • Where to verify: GET /v1/research/constants returns q_table_episodes per domain. If those are 0 for the domains you call, you're in the cold-start regime.

2. te_history backfill

  • What: a rolling buffer of recent omega scores per (tenant, domain) that feeds the change-point and tail-risk detectors (CUSUM/EWMA, Page-Hinkley, jump-diffusion, Lévy flight).
  • Effect: detection layers can flag SUSPICIOUS based on score-history anomalies, which can upgrade recommended_action.
  • Mutates when: each /v1/preflight writes its omega to the bounded rolling buffer.
  • How to opt out: set score_history explicitly in your request and set pg_override_disabled: true. The system uses only your supplied history; no backfill.

3. Severity sticky floor

  • What: if a previous call for the same (tenant, agent_id) returned ASK_USER or BLOCK with an elevated omega, the current call's decision is floored at that previous decision.
  • Effect: recovers from transient noise without flapping between safe/unsafe.
  • How to opt out: set pg_override_disabled: true to skip the previous-decision lookup.

Cosmetic-only drift (does not affect decisions)

free_energy.surprise is normalized against a per-tenant, per-domain maximum that grows with your traffic. It affects warning text only — never recommended_action or omega_mem_final.

How to get strict per-call determinism

For audit, replay, regression-testing, or compliance (NIST AI RMF MEASURE-4.1) scenarios where you need bit-exact reproducibility:

POST /v1/preflight
Authorization: Bearer YOUR_KEY

{
  "memory_state": [...],
  "action_type": "irreversible",
  "domain": "fintech",
  "agent_id": "audit_run_2026_05_03",
  "score_history": [],
  "pg_override_disabled": true
}

What pg_override_disabled: true does: skips the Q-table policy-gradient override, skips the te_history backfill (uses only your score_history), and skips the severity sticky-floor lookup. The response is then fully determined by (memory_state, action_type, domain, score_history, agent_id, _CFG). Restart the process and replay the same request → same omega + same recommended_action.

To verify which config was loaded, fingerprint it:

GET /v1/research/constants
Authorization: Bearer YOUR_KEY

The cfg_checksum field (a sha256 fingerprint of the loaded config) lets you compare against a known-good value. The q_table_episodes field shows the current Q-table state per domain.

What we cannot promise

  • Across process restarts with a different SGRAAL_SCORING_CONFIG — a different config means a different cfg_checksum means different decisions. This is by design (production calibration evolves).
  • Across Sgraal version upgrades — formula changes can shift decision boundaries. We document major changes in the changelog and surface them via cfg_checksum differences.
  • For floating-point precision below ~1e-10 — the decision_trail is byte-exact; raw component values may differ in their last decimal places across CPU architectures.
  • For requests that explicitly query mutable state — endpoints like /v1/agent/{id}/behavioral-profile are intentionally stateful and not in scope of this guarantee.

Compliance mappings

  • NIST AI RMF MEASURE-4.1 — cfg_checksum and q_table_episodes are evidence artifacts; include them in audit packages.
  • EU AI Act Article 13 (Transparency) — the determinism qualification is documented here and surfaced via the response. Customers can produce an audit trail showing same input → same output across replays.
  • GDPR Article 22 (automated decision-making) — per-call replay-determinism enables the "right to explanation": if a customer disputes a decision, you can replay the exact request and trace the decision_trail.
